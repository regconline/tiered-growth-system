## ✅ PHASE 1: MIGRATION AUDIT — ASTRO → NEXT.JS (APP ROUTER)

Go through the **entire codebase file by file** and perform the following checks and fixes:

### 1.1 — Project Structure Verification
- Confirm the project uses the **Next.js App Router** (`/app` directory), not the Pages Router
- Ensure `app/layout.tsx` (or `.jsx`) exists and wraps all pages with proper `<html>` and `<body>` tags
- Verify `app/page.tsx` exists as the homepage entry point
- Check that all route folders follow the App Router convention (`app/about/page.tsx`, `app/services/page.tsx`, etc.)
- Confirm `next.config.js` (or `next.config.ts`) is correctly configured and not using any deprecated Astro-specific options

### 1.2 — Component Migration Checks
- Astro uses `.astro` files — confirm **all `.astro` components have been fully converted** to `.tsx` or `.jsx` React components
- Verify that any Astro-specific syntax (e.g. `---` frontmatter blocks, `<slot />`, `Astro.props`, `define:vars`) has been completely removed and replaced with proper React equivalents (`props`, `children`, `useState`, etc.)
- Check that all **Astro layouts** have been converted to Next.js layouts or reusable React wrapper components
- Confirm no leftover Astro imports like `import { ... } from 'astro'` remain anywhere

### 1.3 — Image Handling
- Astro uses its own `<Image />` component — ensure all images now use **`next/image`** with proper `width`, `height`, and `alt` attributes
- Verify all image paths are correct and images exist in the `/public` directory
- Check that `next.config.js` has the correct `images.domains` or `images.remotePatterns` configured if any external images are used

### 1.4 — Routing & Links
- Astro uses `<a href>` for routing — replace all bare anchor tags used for internal navigation with **`next/link`'s `<Link />` component**
- Verify all dynamic routes use the correct Next.js App Router file-based convention (e.g. `[slug]/page.tsx`)
- Check that `useRouter` imports come from `'next/navigation'` (not `'next/router'`, which is Pages Router only)

### 1.5 — Data Fetching
- Astro fetches data at build time in frontmatter — confirm all data fetching has been migrated to:
  - **`async` Server Components** for static/server-side data
  - **`generateStaticParams()`** for static dynamic routes
  - **`fetch()` with Next.js cache options** (`{ cache: 'force-cache' }` or `{ next: { revalidate: N } }`) where appropriate
- Remove any Astro-specific `getStaticPaths` or `Astro.glob()` patterns entirely

### 1.6 — Metadata & Head Tags
- Astro manages `<head>` tags directly — confirm all pages now use the **Next.js Metadata API**:
  - `export const metadata = { ... }` in Server Components
  - Or `export async function generateMetadata()` for dynamic pages
- Ensure no raw `<head>` or `<Helmet>` tags are being used where the Metadata API should handle it

### 1.7 — Styling Migration
- Check that all CSS, SCSS, or Tailwind styles have been correctly migrated
- Confirm global styles are imported in `app/layout.tsx`
- Verify that any CSS Modules (`.module.css`) are imported and used correctly in React components
- Ensure Tailwind (if used) has `tailwind.config.js` pointing to all correct content paths including `./app/**/*.{ts,tsx}`

### 1.8 — Client vs Server Components
- Identify any components using browser APIs (`window`, `document`, `localStorage`), `useState`, `useEffect`, or event handlers — add `'use client'` directive at the top of each
- Ensure Server Components are NOT importing or using client-only hooks
- Check Context Providers are wrapped in a `'use client'` component and placed appropriately in the layout tree

### 1.9 — Environment Variables
- Astro uses `import.meta.env` — replace all instances with `process.env.NEXT_PUBLIC_*` for client-side variables or `process.env.*` for server-side
- Confirm `.env.local` (or `.env`) exists with all necessary variables and that none of these secrets are committed to source control

### 1.10 — Dependencies
- Run `npm install` (or `yarn`) and verify there are **no peer dependency conflicts**
- Remove any Astro-specific packages (`@astrojs/*`, `astro`) from `package.json` — they serve no purpose and may cause conflicts
- Confirm `next`, `react`, and `react-dom` are all on compatible, latest stable versions
- Check `tsconfig.json` is configured correctly for Next.js (use `"moduleResolution": "bundler"` or `"node"`, `"jsx": "preserve"`, paths alias for `@/` if used)
